Keeping track of technical/architectural decisions made and ideas for what this should do or how it should do it.

* Goal
- mainly to have something nicer than just a Notes app for tracking my workouts
- would be nice to track history as well - e.g. weights done on each date
- shouldn't need any permissions
- background:
  - currently doing a gzcl-style programme customised to what I want - simple, easy to change, general fitness goal rather than e.g. for sports or competition
  - main point is that this has 3 tiers of exercises:
    - t1: low reps, high weights
    - t2: medium reps, medium weights - variations on t1 exercises
    - t3: high reps, lower weights - counting the number of reps you can do (AMRAP As Many Reps As Possible), as well as weight
- Right now I'm focusing less on getting it to a usable state, and more on exploring some technical architecture ideas (like plugins/modules for exercises) and looking at how to use an ECS pattern for this

* Architecture
** Vue
- I just like using Vue
- I think a frontend web framework (like Vue) is nice for a simple app like this
  - as in simple interactions, styling, etc.
  - makes development and debugging easier with web dev tools (can run on browser rather than having to use Android Studio for example)
  - caveat: got more experience with web frontends, there could be an alternative that I just don't know about
** Capacitor
- can use any frontend framework, and has a simple filesystem api
- initial idea was to make a PWA because I figured it'd be simpler than an actual mobile app, but it looked like they don't support the simple filesystem access I wanted
  - see Files section - just want to be able to read/write simple .json files in the filesystem without having to deal with localStorage, indexedDB or anything like that
  - basically I want this app to be fully offline, I don't think that's really the concept for PWAs
- looked into a few alternatives
  - React Native
    - prefer Vue's reactivity system
    - can't just use divs or regular html
  - Flutter, Kotlin Multiplatform
    - render to canvas so can't really make use of web dev tools
  - I think Cordova is essentially an older version of the same concept
** Files
- want to be able to read/write known files without any popups in most cases:
  - read files on startup to get last used workout
  - save when workout is done or just periodically in the background - should not block UI
  - maybe want to be able to export data or use different files for different programmes
- jsons for ease of integration with js
*** exercise.json
- has the exercises with default settings, variations, personal records and notes
- variations inherit properties (e.g. notes) from base, but can override
- e.g. Deadlift is a base, with a main variation at 5x3, and Sumo Deadlift or tier 2 variation at 3x10
- should also show the next weight at sets + reps to lift
  - history.json shows what you last lifted, but I want to store what to do next somewhere
  - this also means we don't need to load history.json most of time, which is good because it's a file that constantly grows
  - I don't think this is the best place to put the 'next' data but I'll have to think about it more
*** programme.json
- has the daily plans - Day 1 (Squat), Day 2 (Overhead Press), etc.
- also things like start date and end date
*** history.json
- records the weights lifted at sets + reps on which date
- this would mostly be wanted for a history view, but it would still need to be updated during workouts after each exercise
  - think about efficient ways to append to .json files?
- something to think about after everything's working

* DEPRECATED Exercises - base and variations
** =Deprecated notes=
- decided to throw this idea of base and variation out and look more into composition with more of an ECS style and using that to implement plugins/modules for exercises
** Would like to define a base exercise and a variation of it
- in my workouts I do the same exercise heavier with fewer reps on one day and lighter with more reps on another day
  - I would like to keep these 2 variations separate but related, mostly for tracking stats
  - also it can be an actual variation on an exercise - like squat one day, hack squat on another
- I would also like to not duplicate data from the base (such as weightIncrement or achieved reps)
  - the variation would just use the base's data if not defined
** Reading and saving
- the variation inheriting the base's data means that the data needs to be combined on read and de-duped on save
- combining isn't a problem, but de-duping on save might be
  - depends on how often it saves
  - right now the 'next' field is in the exercise, so it'll save after every workout
  - will probably want to move 'next' out of exercise, so then it only saves when you make a change to the exercise (probably most often with 'notes')
  - probably only an issue if you have lots of variations - only need to check if it has a baseId and then the objects are pretty small so the comparison probably won't be that expensive
    - still, it's fun to think about these things :)
  - I'm thinking it could be interesting to put the de-duping in a worker or something like that
    - needs to be able to cancel if new data comes in before it's finished processing
    - I think it's fine if the save isn't instant, so long as it's non-blocking
  - another thing to think about is that de-duping doesn't know if it's intentionally inheriting or not (fields might have same value, but might not want to tie them together)
** alternatives to combining right after reading and de-duping just before saving
- only combine when reading in e.g. the ExerciseItem (pushing things closer to the viewing/modifying rather than the reading/writing)
  - wouldn't need to de-dupe, but this does mean that everything needs to know about variations
  - I think it's better to have this kind of thing closer to read/write, so nothing else needs to care about the implementation
- perhaps the ExerciseVariation should have a list of all the fields that are inherited?
  - more explicit
  - then de-duping isn't needed
  - this would mean that when the base updates a field, it would need to update all the variations that reference it
  - there could be a data mismatch or de-sync between the base and variation - maybe could update/check manually or on load?
** alternative patterns
- maybe look into more of an ECS pattern?
  - entity: exercise
  - components: sets, reps (then can make a system for going to the next tier)

* Modularity / plugins with an Entity Component System approach
** What plugins would have
- data to add to exercise entity
  - e.g. sets/reps: a list of sets + reps to use (5x3, 6x2, 10x1)
- how to render in an ExerciseItem or other components
  - e.g. sets/reps: rendering text "5x3", buttons to increment/decrement sets, etc.
- any logic the plugin needs
  - e.g. sets/reps: button to go to the next sets/reps after failure (5x3 -> 6x2)
- would need to save data for an exercise
- might need to think about how they would interact with other plugins in an exercise
** Exercise entity
- has some basic fields like name or id
- maybe should have a group field? (for the tier 1 and tier 2 exercise variations)
- other fields come from components which could be added based on what you want for that exercise
  - e.g. you might use a component that just shows sets and reps or a component for AMRAP that shows reps achieved
** Sets + reps component
- a list of sets/reps (e.g. 5x3, 6x2, 10x1) as part of the data for this plugin
- system for going to the next sets/reps (e.g. 5x3 -> 6x2) when you press a button
- render setsXreps in ExerciseItem, show current set you're on, and buttons for incrementing/decrementing the sets
** AMRAP component
- shows the achieved reps for an exercise at that weight
  - e.g. an exercise could be 3x15+ and you'd want to record that you did 17 reps
** weight component
- shows weight and has a weightIncrement so you press a button and it'll increase the weight by that amount
- should also be able to explicitly set the weight as well
** notes component
- render/edit notes for an exercise
- I'm thinking that I still like the idea of combining notes for exercise variations
  - +could use an exercise group field?+
    - although how would you know what the base notes are vs notes just for this exercise?
  - could have something like a noteId, so multiple exercises could reference the same noteId
    - changing notes for a noteId in one exercise updates the notes in any other exercise using that noteId
    - notes would be a list, so you can select a pre-existing note (i.e. use an existing noteId) or make a new one

* Saving current workout progress to a temp file
- probably want to save e.g. which set you're on to a temp file, to make sure it's easy to resume in case you close the app
- saving for this does need to be very fast
- keep track of:
  - which set you're on for each exercise
  - current exercise day
  - which exercises are open
